## Shared Rules For Planned APIs

- Every API still marked **Planned / Not Implemented** below is unavailable until it appears in generated OpenAPI.
- All business APIs use `/api/v1` and the shared error shape with `code`, `message`, `details`, and `request_id`.
- List APIs must support backend pagination with `page` and `page_size`; add filters/sort where listed.
- Workflow mutations use `expected_version` when they update state; the frontend must handle `409` by refetching.
- `400` is validation, `401` is unauthenticated, `403` is forbidden, `404` is not found or out of scope, `429` follows `Retry-After`, and `500` should show support-friendly `request_id`.
- Money values are decimal strings. Dates/times use ISO formats. UUIDs are strings.
- Documents and downloads must go through backend document APIs; frontend must not access object storage directly.
- Expense additions must keep Manager -> Finance terminology and must not add legacy approval roles back into active flows.

## Frontend Integration Guidance

- Use `openapi.json` and `ENDPOINT_INDEX.md` only for generated clients and live API calls.
- Treat this report and `FRONTEND_BACKEND_GAP_AUDIT.md` as backlog input, not runtime contract.
- Keep mock/local state for P2-P4 modules until their APIs are implemented, tested, and regenerated into OpenAPI.
- For P1 endpoints, frontend adapters should tolerate absent optional fields until backend expansion lands.
- Do not hard-code planned paths in production UI. Feature flags or integration TODOs are acceptable while backend work is pending.
- After each backend phase: update implementation, add tests, regenerate OpenAPI, regenerate `ENDPOINT_INDEX.md`, then update this report from Planned to Implemented/Changed.
